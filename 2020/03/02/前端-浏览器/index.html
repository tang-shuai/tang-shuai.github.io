缓存

  • 缓存介绍

    缓存分为强缓存和协商缓存,流程图

    缓存机制相关的字段都是在请求和响应头上

    请求头:

  • 强缓存

    强缓存，在缓存有效期内，客户端直接读取本地资源

    强缓存返回状态码是200

    • Expries

      在http1.0 中使用，表示资源失效的具体时间点 Expires:Sat, 09 Jun 2018 08:13:56 GMT。若访问器和本地时间不一样，在现在的http 1.1 中换成了max-age

    • Cache-contol

      指定指令来是先缓存

      • max-age: 强缓存的有效时间，单位秒max-age=30672000
      • no-cache: 不使用本地缓存。需要使用缓存协商，先于服务器确认返回的响应是否被，如果之前的响应中存在ETag，那么请求的时候会与服务端验证，如果未被更改，则可以避免重新你下载
      • no-store: 直接禁止浏览器缓存数据，每次用户请求该资源，都会向服务端发送一个 请求，每次都会下载完整的资源
      • public: 可以被所有的用户缓存，包括用户和CDN等中间代理服务器
      • private: 只能被终端用户的浏览器缓存，不允许CDN等中继缓存服务器对其缓存

  • 协商缓存

    • 协商缓存，在使用本地缓存之前，需要先跟服务器做个对比，服务器告知你的资源可用，就可以直接使用本地资源。反之，服务器返回最新的资源给客户端，客户端收到后会更新本地资源

      • 状态码：
        • 若本地资源是最新的，返回304.
        • 重新获取资源，就正常返回200

    • Last-modified If-Modified-Since

      采用资源最后修改时间来判断，单位精确到秒

      Last-Modified: 服务器资源的最新更新时间Tue,14 Jan 2020 09:18:29 GMT

      If-Modified-Since:发起协商，把本地的记录文件更新时间传给服务器，服务器进行判断比较

      这个判断方式是http 1.0的产物,精确到秒，若文件的更新频率在秒级以内，就会出现文件不一致

    • ETag If-None-Match

      为了解决上面的问题，http 1.1加了这组标记

      ETage: 服务器根据内容生成唯一的字符串标识

      If-None-Match: 发起协商，把本地的hash标识传给服务器，服务器进行判断比较

  • 浏览器缓存存放位置

    缓存的两个来源：

    • memory cache:内存中读取
    • disk cache: 硬盘中读取

PWA

PWA(渐进式Web应用)运用现代的web API以及传统的渐进式增强策略来创建跨平台Web应用程序需要专门学习有利于页面性能优化

  • App Shell

    App Shell 架构是构建Progressive Web App 的一种方式，这种应用能可靠且及时的加载到您的用户屏幕上

    这个模型就是将资源离线缓存，可以确保重复访问都有快速响应的特性，页面可快速的渲染，网络仅仅获取数据

  • ServiceWork

    PWA 的核心，缓存可以让页面尽快的加载，但是必须在有网络的情况下才行，没网络只能使用离线缓存，也就是ServiceWork

    ServiceWorker 有以下的功能和特性：

    • 一个独立的worker 线程，独立鱼当前网页进程，有自己独立的worker context
    • 一旦被install ,就永远存在，除非手动删除unregister
    • 用到的时候可以直接唤醒，不用的时候自动睡眠
    • 可编程拦截代理请求和返回，缓存文件，缓存的文件可以被网页进程取到（包括网络离线状态）
    • 离线内容开发者可控
    • 能向客户端推送消息
    • 不能直接操作DOM
    • 必须在HTTPS环境下才能工作
    • 异步实现，内部大都是Promise 实现

    js 是单线程的，ServiceWork 独立线程意味着不会阻塞js执行；可编程拦截代理请求和返回，可自定义文件缓存策略

这些特点意味着开发者有足够的权限去操作缓存，让缓存做到优雅，效率达到极致

设计缓存策略：

  • 缓存优先，先查询缓存，若存在，直接返回，不存在，请求服务，更新缓存

  • 服务端有限，不查询缓存，直接请求服务端，服务端失败才会查询缓存

  • 稳定优先，先查询缓存，有就读取，同时请求服务端更新资源

重绘和回流(重排)

html文档和css渲染过程的图：

重绘： 对元素的外观做处理

回流： 页面采用流式布局来绘制，当一个节点空间属性发生变化，会影响到其他节点的空间布局，需要重新收集节点信息，在进行绘制。

所以回流一定会触发重绘

  • 回流的优化

    对树的局部甚至全局重新生成是非常耗性能的，所以要避免频繁触发回流

    • 现代浏览器已经帮我们做了很多的优化，采用队列存储多次的回流操作，然后批量执行，但获取布局信息除外，因为要获取实时的数值，浏览器就必须要清空队列，立即执行回流
    • 避免连续多次修改，可以合并修改，一次修改
    • 对于大量不同得DOM修改，可以优先脱离文档流使用定位，或者display：none ,在文档流外修改完成后再放回文档中
    • 通过节流和防抖控制触发频率
    • css3硬件加速，transform,opacity,filters,开启后，会重新渲染层

  • 开启GPU加速的方法

    开启后，会将dom 元素提升为独立的渲染层，它的变化不会在影响文档流中的布局

    • transform：translateZ(0)
    • opacity
    • filters
    • Will-change

Http和Https协议

http协议

http 是建立在TCP 上的应用层协议，曹文本传送协议

  • http1.0:客户端的每次请求都要求建立一次单独的连接，在处理完本次请求后自动释放连接
  • http1.1:可以在一次连接中处理多个请求，并且多个请求可以重叠进行，不需要等待一个请求结束后在发送下一个请求
  • http2.0:可支持多路复用，一个tcp 可同时传输多个http 请求，头部数据还做了压缩

TCP

tcp 是传输层协议，特点是：三次握手和四次挥手

  • 三次握手建立连接：

    • 客户端发送位码为syn=1，随机产生数据包到服务器，服务器有SYN=1知道客户端要求建立联机(客户端：我要链接你)
    • 服务器收到请求后要确认联机信息，向A发送ack number=(客户端的seq+1)，syn=1，ack=1随机生成数据包（服务器：好的，你来链接吧）
    • 客户端收到后检查ack number是否正确，即第一次发送的seq number+1,一级位码ack是否为1，若正确则建立连接

  • 四次挥手断开连接的过程：

    • 客户端发送请求给服务端，申请主动断开连接，进入等待状态，不在往服务端发送数据，但是接受数据（客户端：我要断开连接了）
    • 服务端收到后，告知客户端知道了，服务端进行等待状态，不在接收数据，但是可以继续发送数据（服务端：好，我知道了，但是要等一等）
    • 客户端收到服务端的告知后，进入下一阶段的等待（客户端：好，我等）
    • 服务端完成剩余数据的发送后，告知客户端可以断开了，服务端不接收和读取数据（服务端：你可以断开了）
    • 客户端收到后，告知服务端，已收到，然后释放链接（客户端：好的，我断开了）
    • 服务端收到后，也释放链接

UDP

传输层的另外一个协议UDP被称为用户数据报协议，无连接的传输协议

UDP 是报文的搬运工，不需要建立完全可靠的链接，不保证数据的可靠性，因为协议控制项比较少，且报文头部简单，报文体积相对较小，速度上相比更快，实时性更高，比如电话会议，多媒体数据流才用的就是UDP。

https

http 报文传输过程中是铭文的，可以通过抓包的方式看到报文内容，这就暴露一个安全问题，为了解决这个问题，就有了TLS ,https = http+TLS

TLS: 安全传输层协议，用于在两个通信应用程序之间提供保密性和数据的完整性，该协议由两层组成：TSL 记录协议和TLS 握手协议

TLS利用非对称加密演算来对通信方做身份的认证，之后交换对称密钥作为会谈的密钥（Sessionkey),因此https分成两个阶段

  • 通过非对称加解密确认对方身份是否合法，若合法生成会话密钥
  • 报文的在发送前，先用会话密钥进行对称加密在传输

TLS 握手

步骤如下：

  • 客户端请求服务端建立SSL 链接，服务端并向客户端发送一个随机数randomC 和CA 机构颁发的证书
  • 客户端对证书进行验证，验证通过后，生成一个随机数randomS，用公钥对randomS 加密，同时用randomS 生成一段签名，发送给服务端
  • 服务端接收到后，用私钥对密文解密，用解密后的key生成签名，并对客户端传来的签名进行比较，检验通过后，然后生成一个随机数randomP ,并对私钥加密，还有随机数生成的hash值，一并发送给客户端
  • 客户端用公钥解密，并验证 hash通过后，两端利用 randomC randomS ramdomP 通过一定的算法生成session key ，后续的报文将通过session key 对称加密进行传输

前端安全规范

前端安全主要分为XSS和CSRF

XSS

跨站脚本攻击，攻击者将一段可执行的代码注入到网页中，如链接，输入框，分为持久性和临时性的，持久性的是恶意代码被存储到数据库中，会造成持久的攻击；临时性的是仅在当前被工具页面上生效

防范方式：对与网页上获取的内容要做转义处理

CSRF

跨站请求伪造，构造一个钓鱼网站，利用站点对浏览器的信任，从而欺骗用户，发起恶意操作

用户在浏览器登录后，站点是信任浏览器的，但浏览器是没法知道请求是否是用户自愿发起的，站点信任后，所发起的请求浏览器都是信任的。

那么用户在已登录的情况下，钓鱼站点中发起跨域请求，跨域标签或者form表单，就会把用户的认证信息cookies带上，从而达到伪造用户身份进行攻击

防范方式：

  • 服务端校验Referer，但某些浏览器可能可以修改Referer
  • 随机token，每访问页面就生成一个token，页面中的请求把这个token带上，服务端对token进行校验，注意这个token不能存储在cookie中
